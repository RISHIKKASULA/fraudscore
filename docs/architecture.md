# fraudscore — Architecture & Design

One-liner: a calibrated card-fraud scoring service that makes expected-cost decisions —
review a transaction when p̂ × amount exceeds the cost of reviewing it — served over FastAPI,
evaluated in dollars with confidence intervals, not in accuracy.

Deviations from this document require an ADR entry in [decisions.md](decisions.md);
simplest defensible choice wins.

---

## 1. Data

- **Dataset:** ULB credit-card fraud (Kaggle `mlg-ulb/creditcardfraud`). 284,807 transactions,
  492 fraud (0.172%). Columns: `Time` (seconds since first transaction; span ≈ 172k s ≈ 2 days),
  `V1..V28` (PCA-anonymized), `Amount`, `Class` (1 = fraud).
- **Not committed to git.** `scripts/fetch_data.py` downloads via the Kaggle CLI to `data/raw/`
  (gitignored) and verifies row count + SHA-256.
- **Committed fixture:** `data/fixtures/synthetic.csv` — ~2,000 rows, same schema, inflated
  fraud rate (~5%), generated by `scripts/make_fixture.py`, seeded (42), deterministic. CI runs
  the full pipeline on this; the real data never enters CI.
- **Split: chronological.** Sort by `Time`: train 60% / calibration 20% / test 20%.
  **Rationale: leakage avoidance — train on the past, decide on the future, exactly as the
  service would run in production.** The dataset spans only ~2 days, so no concept-drift claims
  are made; a random split would simply let the model peek at the future side of transaction
  sequences it will be judged on. Honest even though it may cost a little metric shine.

## 2. Features (`src/fraudscore/features.py`)

sklearn `Pipeline` + `ColumnTransformer`, serialized inside the model artifact so serve-time
transforms are identical to train-time by construction:

- `V1..V28`: passthrough (already PCA components).
- `Amount`: `log1p`, then `RobustScaler` (heavy right tail).
- `Time` → **`cycle_phase`**: `phase = (Time % 86400) / 86400`, encoded as
  `phase_sin = sin(2π·phase)`, `phase_cos = cos(2π·phase)`. Drop raw `Time` (sequence index;
  raw use would leak the chronological split).
  **Naming honesty:** `Time` is seconds since the *dataset's first transaction*, not clock
  time, so this feature is *phase within a 24-hour cycle relative to dataset start* — it can
  capture daily periodicity but is NOT time-of-day. Code, docstrings, eval report, and README
  all use this description; no "hour of day" or "nighttime fraud" language anywhere.

**Class imbalance — the consistent story:** no resampling and **no class weights, anywhere**.
SMOTE/oversampling distorts the base rate and poisons calibration; class weights distort
predicted probabilities the same way. Both models are trained plain, then calibrated, and the
decision layer reasons in expected dollars — imbalance is handled where it belongs, at the
decision, not by lying to the model about the base rate. (Roadmap: test that claim against a
SMOTE variant and a class-weighted variant on expected cost; hypothesis: they don't win.)

## 3. Models

- **Baseline:** `LogisticRegression(max_iter=5000, random_state=42)` — plain, no class
  weighting (see §2). It's a calibrated-and-costed baseline, not a strawman: it goes through
  the identical calibration and decision layers and stays in the eval report permanently.
- **Main:** `HistGradientBoostingClassifier(random_state=42)`, small grid
  (`max_iter ∈ {200, 400}`, `learning_rate ∈ {0.05, 0.1}`, `max_leaf_nodes ∈ {31, 63}`)
  selected by PR-AUC via 3-fold `TimeSeriesSplit` **within the train window only**.
  Chosen over XGBoost deliberately: one fewer dependency, native sklearn integration, trains
  in seconds on a laptop — the differentiator here is the decision layer, not the model brand.

## 4. Calibration (`src/fraudscore/calibrate.py`)

- Fit on the **calibration split only** (`cv="prefit"` pattern), never on train or test.
- **Both isotonic and sigmoid (Platt) fitted.** ~98 fraud cases is thin for isotonic (tail
  overfit risk); sigmoid is the expected winner, but it's an empirical question.
- **Selection rule:** lower Brier score under 5-fold CV within the calibration split;
  tie → better reliability fit in the p < 0.1 region. Both curves ship in the eval report.
- Calibration matters doubly here: the primary decision rule multiplies p̂ by dollars, so a
  2× miscalibration is a 2× mispricing of risk.

## 5. Cost model & decision policy (`src/fraudscore/cost.py`) — the heart of the project

**Cost matrix per transaction *i* with amount *aᵢ*:**

| Outcome | Meaning | Cost |
|---|---|---|
| TN (approve legit) | normal business | 0 |
| FP (review legit) | analyst review + friction | `c_review` = **$10** (default, configurable) |
| TP (review fraud) | fraud caught, review still costs | `c_review` = $10 |
| FN (approve fraud) | charge-back eaten | **aᵢ** |

**PRIMARY — amount-aware expected-cost rule (Bayes-optimal under this matrix):**

```
review  ⟺  p̂ᵢ · aᵢ ≥ c_review
```

Expected cost of approving = p̂ᵢ·aᵢ; of reviewing = c_review; pick the cheaper action per
transaction. No fitted threshold — the "threshold" is the cost ratio itself, which is why the
rule stands or falls with calibration quality. Consequence worth owning: a transaction under
$10 is never reviewed even at p̂ = 1 — economically correct under this matrix, and the README
limitations section says so explicitly (real systems layer velocity/fraud-ring logic on top;
roadmap).

**BASELINE — single global threshold t\*:** empirical cost curve on the calibration split,
`cost(t) = Σᵢ [p̂ᵢ ≥ t]·c_review + [p̂ᵢ < t]·yᵢ·aᵢ` over a dense grid, t\* = argmin, frozen
before touching test. Kept because it's what most shops actually deploy — the dollar gap
between it and the amount-aware rule *is the finding*.

**Reported comparisons (test split, dollars per 10k transactions — the headline numbers):**
1. amount-aware rule vs single-threshold t\* (both calibrated) — the headline
2. t\* vs naive t = 0.5 on the uncalibrated model — what a notebook would ship
3. calibrated vs uncalibrated under the amount-aware rule — what calibration buys in dollars
4. everything vs "approve all" (the do-nothing floor)

**Uncertainty:** ~98 test frauds make point estimates noisy. Every reported cost and every
% improvement carries a **bootstrap 95% CI**: resample test-split rows with replacement,
B = 10,000, seeded (42), percentile intervals; vectorized (a B×n index matrix is cheap).
Report format everywhere, including README: `point [CI_low, CI_high]`. If an improvement's CI
includes zero, we say so plainly rather than hiding it — that honesty is itself part of the
project's brand.

`cost_params.yaml` at repo root holds `c_review`, the grid, and bootstrap settings, so a
reviewer can rerun with their own economics — the framework is the point, not the $10.

## 6. Evaluation & artifacts (`src/fraudscore/evaluate.py`)

`fraudscore evaluate` (CLI) writes `docs/eval-report.md` + PNGs (matplotlib):
- PR-AUC (primary), ROC-AUC (secondary, footnoted as inflated under imbalance), Brier
- Reliability diagrams (uncalibrated / sigmoid / isotonic overlay)
- Expected-cost-vs-threshold curve with t\* marked, plus the amount-aware rule's cost drawn as
  a horizontal reference line — the signature chart: a curve that never reaches the line
- The four dollar comparisons of §5, each as `point [95% CI]`
- Confusion matrices at t\* and under the amount-aware rule (test split)
- Data card: split boundaries, base rates per split, row counts

`fetch → train → evaluate` reproduces the report bit-for-bit (seeded, bootstrap included).
Model artifact (`model.joblib` + `model-card.json`: version, git SHA, metrics + CIs, t\*,
cost params, trained-at) is gitignored; the report is committed.

## 7. Serving (`src/fraudscore/serve.py`)

FastAPI + pydantic v2, uvicorn. Artifact loaded once at startup.

```
POST /score
  request  { "amount": 149.62, "time": 40632.0, "v": [28 floats] }      # extra fields → 422
  response { "fraud_probability": 0.0031,          # calibrated
             "expected_fraud_cost": 0.46,          # p̂ · amount, the decision quantity
             "decision": "approve",                # expected_fraud_cost ≥ c_review → "review"
             "decision_rule": "expected_cost",
             "c_review": 10.0,
             "model_version": "1.0.0",
             "scored_at": "2026-07-15T18:22:04Z" }

GET /health      → { "status": "ok", "model_loaded": true }
GET /model-info  → model-card.json (incl. baseline t*, so both rules are inspectable)
```

Validation: `amount ≥ 0`, `time ≥ 0`, `len(v) == 28`, all finite; violations → 422 with
field-level detail. Batch: `fraudscore score-batch in.csv --out scores.duckdb` appends to a
DuckDB `scores` table (id, p̂, expected_fraud_cost, decision, model_version, scored_at).

## 8. Test plan (`tests/`)

- **Unit — features:** transform determinism; `cycle_phase` hand-checked (Time=0 → sin 0/cos 1;
  Time=21600 → sin 1/cos ≈ 0; Time=86400 ≡ Time=0); log1p+robust-scale on known vector; no-NaN
  guarantee on fixture.
- **Unit — cost:** amount-aware rule on a hand-computed 5-transaction toy case (expected
  decisions as literals, including a p̂=0.9/$5 "never review" case and a p̂=0.02/$800 "review"
  case); `cost(t)` toy table; t\* picks the known argmin; degenerate cases (all-legit, all-fraud).
- **Unit — bootstrap:** seeded determinism (same seed → same CI); CI contains point estimate;
  rigged two-value case yields hand-checkable percentiles.
- **Unit — calibration:** ranking preserved (spearman ρ = 1 vs raw scores); probabilities in
  [0,1]; selection rule picks lower-Brier method on a rigged synthetic case.
- **Integration:** end-to-end on the synthetic fixture — fixture → train → calibrate → evaluate
  → report exists and parses; CI-runnable in ~2 min (bootstrap B reduced via config in CI).
- **API contract:** TestClient — happy path schema incl. `expected_fraud_cost`; 422s (missing
  field, 27 floats, negative amount, non-finite); /health; /model-info matches artifact;
  decision flips across the c_review boundary (same p̂, amounts straddling break-even).
- **Regression:** golden metrics from the synthetic-fixture run (PR-AUC, Brier, t\*, one CI
  endpoint pair, tolerance ±1e-6). Real-data metrics live only in the committed eval report,
  asserted nowhere.

## 9. Acceptance criteria for v1.0.0

- Quickstart in README works on a clean machine (verified once in Docker)
- Eval report committed: all four §5 comparisons as `point [95% CI]`, signature chart included
- README headline numbers carry their CIs; if a CI includes zero, the README says so
- CI green; unit + contract + integration + regression tests passing; seeded end-to-end
  reproducibility including bootstrap
- p50 /score latency < 50 ms locally (uvicorn, single worker) — measured, in README

## 10. README limitations section must include

(a) reviewed fraud is assumed always stopped — review cost is charged but FN cost is fully
avoided, which is optimistic about analyst performance; (b) transactions below `c_review` are
never reviewed under the primary rule — correct under this cost matrix, unsafe against fraud
rings without velocity logic; (c) `cycle_phase` is periodicity relative to dataset start, not
time-of-day; (d) anonymized PCA features limit feature-engineering claims.
